# Cipher v1.1.1 — donations + dev quality-of-life

Patch release on top of [v1.1.0](https://github.com/Oykdo/cipher/releases/tag/v1.1.0). Same privacy contract, same E2E protocol, same binary surface — but Stripe donations now actually work end-to-end, and `npm run dev` no longer hangs on a backend timeout.

## What's new since v1.1.0

- **Stripe donations are persisted**. The `/api/public/stripe/webhook` handler now upserts every `checkout.session.completed` into a new `donations` table, then transitions status on `payment_intent.succeeded` / `.payment_failed` / `charge.refunded`. Idempotent on `stripe_session_id` — Stripe retries (which are routine and expected) never duplicate rows.
- **No PII in the new table**. `donations` records `stripe_session_id`, `stripe_payment_intent_id`, `amount_cents`, `currency`, `status`, `customer_email` (Stripe-required for refunds), `metadata` jsonb. **No `user_id`, no IP, no UA.** Donations are business records, not surveillance — see migration 005 header for the rationale.
- **9 new vitest tests** lock the persistence behaviour: signature verification, idempotency, status transitions, and the 503 / 500 error paths. Gated behind `DATABASE_URL_TEST` so a casual `npm test` stays cheap.
- **Electron dev backend timeout fixed**. `startBackend()` was duplicating the bridge in `npm run dev` (concurrently already runs `dev:bridge`), and the duplicate was racing with the primary on `initializePsiServer` and losing. Now both dev and packaged installs default to **hosted-bridge mode**, with an opt-in via `CIPHER_BUNDLED_BRIDGE=true` for self-hosted use cases.
- **Local Stripe testing guide** at [`docs/STRIPE_LOCAL_TESTING.md`](https://github.com/Oykdo/cipher/blob/master/docs/STRIPE_LOCAL_TESTING.md) — full Stripe CLI loop documented (install → login → listen → trigger → verify the row in `donations`).
- **`.env.example` cleaned up**. No more `sk_live_*` template that nudged people to paste real production keys locally; the file now explicitly says live keys belong only in `fly secrets`.

## Privacy contract — still upheld

- **Privacy contract first**. Read [`CIPHER_PRIVACY_GUARANTEES.md`](https://github.com/Oykdo/cipher/blob/master/CIPHER_PRIVACY_GUARANTEES.md) — it's the promise, and the code keeps it.
- **Mnemonic stays 100% on device**. BIP-39 generation in the browser, never transmitted.
- **No server-side cleartext**. `messages.body` only contains opaque E2E ciphertext. The sender keeps their own copy via a self-addressed envelope (e2ee-v2).
- **Dual-TTL retention**. Delivered messages are deleted within 7 days. Undelivered ones are dropped after 30 days.
- **Zero PII in logs**. Pino redacts IP, user-agent, cookies, auth headers.
- **No persistent audit logs**. The `audit_logs` table was dropped in migration 004.
- **Client-side time-lock via tlock + drand**. The server is no longer the oracle.

## What works in this release

Everything from v1.1.0:
- Signup via BIP-39 mnemonic + local password, quick-unlock, 1-to-1 E2E (X3DH + Double Ratchet), encrypted attachments, burn-after-reading, time-lock, local backup export.

Plus:
- **Donations end-to-end** — UI button → Stripe Checkout → webhook → `donations` row.

## What does NOT work yet

- Group conversations (schema supports it, per-recipient delivery tracking deferred).
- Push notifications (blocked on the upcoming mobile companion app).
- Code signing (binary unsigned — Windows SmartScreen will warn; click "More info" → "Run anyway").
- Mac/Linux builds (build-from-source only, until the CI matrix is in place).

2. Verify the SHA-256 against `SHA256SUMS.txt`
3. Double-click, allow Windows SmartScreen, install, launch

### macOS / Linux

Build from source per the [README](https://github.com/Oykdo/cipher/blob/master/README.md). Pre-built downloads will come once the CI matrix is wired.

## Architecture in brief

- Frontend: React + Vite + libsodium + Double Ratchet (X3DH).
- Bridge: Fastify + PostgreSQL (Neon) + Socket.IO, deployed on Fly.io (Paris, scale-to-zero).
- All user secrets live in a local KeyVault sealed by PBKDF2-SHA256 600k of the user's password. The server only sees opaque ciphertexts and public keys.

## Verify it yourself

Clone the repo, build, compare the hash. Or run the donations test suite against your own test DB:

```bash
cd apps/bridge
DATABASE_URL_TEST=postgresql://... npx vitest run src/__tests__/stripe-donations.test.ts
```

## Migrating from v1.1.0

If you ran v1.1.0 against your own bridge: apply migration `005_add_donations.sql` against your Postgres before launching v1.1.1, otherwise the webhook handler will return 500 on every Stripe event.

```bash
psql $DATABASE_URL < apps/bridge/scripts/migrations/005_add_donations.sql
```

The hosted bridge on Fly is already migrated.

## Known caveats

- Bridge in Paris region (Fly `cdg`), Neon DB in US East — ~80 ms latency per request. Migration from Neon to Frankfurt is planned.
- If you install on a device without a previous Cipher dev install, the `pwd_<username>` localStorage and KeyVault IndexedDB are created from scratch — no migration from another install.
