# Cipher v1.0.0 — privacy-l1 alpha

First public ship. Cipher tient sa promesse de fond : **le serveur ne stocke rien qu'il n'ait à stocker**, c'est documenté dans un contrat public ([`CIPHER_PRIVACY_GUARANTEES.md`](../CIPHER_PRIVACY_GUARANTEES.md)) et vérifié par 4 invariants automatisés en CI.

## Highlights

- **Privacy contract first**. Lis [`CIPHER_PRIVACY_GUARANTEES.md`](../CIPHER_PRIVACY_GUARANTEES.md) — c'est la promesse, et c'est le code qui la tient.
- **Mnémonique 100 % côté device**. Génération BIP-39 dans le browser, jamais transmise. Le serveur ne sait pas la deviner même s'il est compromis.
- **Pas de cleartext serveur**. `messages.body` ne contient que du ciphertext E2E opaque. Le sender garde sa propre copie via un envelope auto-adressé (e2ee-v2).
- **Dual-TTL retention**. Les messages livrés sont supprimés sous 7 jours. Les non-livrés sont droppés après 30 jours. Pas d'historique côté serveur.
- **Zéro PII dans les logs**. Pino redacte automatiquement IP, user-agent, cookies, auth headers.
- **Pas d'audit logs persistants**. La table `audit_logs` est dropped (migration 004). Un ring buffer mémoire bornée remplace pour le debug d'incident actif.
- **Time-lock client-side via tlock + drand**. Le serveur n'est plus l'oracle.

## What works in this release

- Inscription via mnémonique BIP-39 (12 ou 24 mots) + password local
- Quick-unlock (PBKDF2-SHA256 600 000 itérations, sealed-by-password KeyVault)
- Conversations 1-to-1 chiffrées E2E (X3DH + Double Ratchet)
- Pièces jointes chiffrées
- Burn-after-reading (le serveur supprime vraiment, pas un placeholder)
- Time-lock (chiffrement drand)
- Export local du backup (mnémonique lue depuis le KeyVault, jamais du serveur)

## What does NOT work yet

- Conversations de groupe (le schéma supporte mais le tracking de delivery par destinataire est différé)
- Push notifications (bloqué sur le mobile companion app à venir)
- Stripe pour donations CB (UI présente, backend pas encore configuré)
- Code signing (binaire non-signé — Windows SmartScreen va warner ; cliquer "More info" → "Run anyway")
- Builds Mac/Linux (build-from-source uniquement le temps que la CI matrix soit en place)

## Install

### Windows

1. Download `Cipher-Setup-1.0.0.exe`
2. Verify the SHA-256 against `SHA256SUMS.txt`
3. Double-click, allow Windows SmartScreen, install, launch

```
SHA-256: 2bbd5249d7ae8d9aca5bc2aef49b3c18d7e1e9396a2629d97d09bb234d39317e
Size:    96 422 360 bytes (≈ 92 MiB)
```

### macOS / Linux

Build from source per the README. Pre-built downloads come once the CI matrix is wired (see issue tracker).

## Architecture en bref

- Frontend : React + Vite + libsodium + Double Ratchet (X3DH).
- Bridge : Fastify + PostgreSQL (Neon) + Socket.IO, déployé sur Fly.io (Paris, scale-to-zero).
- Tous les secrets utilisateurs vivent dans un KeyVault local sealed par PBKDF2-SHA256 600k du password de l'utilisateur. Le serveur ne voit que des ciphertexts opaques + des clés publiques.

## Verifier soi-même

Cloner le repo, builder, comparer le hash. Ou lancer les invariants tests sur ta propre DB de test :

```bash
cd apps/bridge
DATABASE_URL_TEST=postgresql://... npm run test:invariants
```

- Bridge en région Paris (Fly `cdg`), DB Neon en US East — ~80 ms de latence par requête. Migration de Neon vers Frankfurt à planifier (cf. `INFRA_NOTES.md`).
